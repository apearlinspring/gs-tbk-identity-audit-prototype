# 项目表述与项目定位

本文用于沉淀当前项目的对外展示表述。核心原则：同一个项目可以按 AI（Artificial Intelligence，人工智能）应用开发或 AI 安全两个方向讲，但不能把工程原型夸大成已上线生产系统、模型训练算法或 Prompt Injection（提示词注入）完整防护系统。

## 通用项目名

基于 Rust（系统级编程语言）+ FISCO BCOS（金融区块链合作联盟开源区块链底层平台）的分布式身份监管与链上审计工程原型

## 一句话项目描述

围绕“匿名使用、行为留痕、异常追责、多方监管”目标，整理并验证 1 个 Proxy（代理）、4 个 Node（管理员节点）、2 个 User（用户）的多角色 E2E（End-to-End，端到端）流程，实现身份字段 CL（Castagnos-Laguillaumie，同态加密方案）加密/ZKP（Zero-Knowledge Proof，零知识证明）、签名与身份密文上链、链上历史查询、恶意签名 Verify/Open（验证/揭示）和只读 Web（网页）/API（Application Programming Interface，应用程序接口）审计控制台展示。

## 第三方测试记录版本

适合作为项目页或个人作品集中的摘要版本，重点突出“真实项目 + 第三方功能测试用例 + 多服务器环境”，但不把项目夸大成完整生产系统。证据索引见 [third-party-test-evidence.md](third-party-test-evidence.md)。

可写项目描述：

> 参与研发一套面向许可链的分布式身份监管系统。系统基于 GS-TBK（Group Signatures with Time-bound Keys，带时间绑定密钥的群签名方案）群签名方案与 FISCO BCOS 区块链，通过委员会节点的阈值密钥生成、用户匿名签名、链上身份数据登记和异常身份揭示机制，在保护用户隐私的前提下实现恶意用户追责。项目在 5 台云服务器和 1 台本地服务器环境中完成第三方功能测试用例验证，覆盖阈值密钥生成、用户注册、签名上链、身份揭示、隐私数据密文上链和解密校验等流程，相关测试结论均为通过。

可选 bullet：

- 负责身份字段处理与 CL 同态加密模块，实现用户身份字段编码拼接、CL 加密、承诺和 ZKP 生成，将身份密文、承诺和证明数据通过智能合约写入 FISCO BCOS，并支持监管侧链上读取、解密验证和字段还原。
- 负责多节点部署联调和测试记录整理，在 4 个委员会节点、1 个代理节点和本地用户节点环境中跟踪 KeyGen（密钥生成）、用户注册、签名上链、链上查询、Verify/Open 身份揭示等流程，支撑第三方功能测试用例通过。
- 当前仓库进一步将历史测试材料整理为可复现工程原型，完成真实 VM（Virtual Machine，虚拟机）上的 1 Proxy + 4 Node + 2 User 生产化 smoke、链上审计查询、恶意揭示摘要和只读 Web/API 审计控制台展示。

## AI 应用开发版本

适合对外展示偏应用交付、后端工程、AI 平台、Agent（智能体）应用、工具调用审计和企业内部平台的方向。重点突出系统集成、可演示 Web/API 入口、数据闭环和工程边界。

可写项目描述：

> 基于 Rust + FISCO BCOS 搭建分布式身份监管与链上审计工程原型，完成多角色协议运行、身份密文与签名上链、历史审计查询、异常行为揭示和只读 Web/API 审计控制台；该能力可迁移到 AI 应用中的工具调用留痕、敏感操作追责和模型服务审计场景。

可选 bullet：

- 负责将历史 demo（演示工程）整理为 Rust workspace（工作区）工程原型，收口 Proxy、Node、User 多角色启动入口和运行证据，支持 1 Proxy + 4 Node + 2 User 的生产化 smoke（冒烟验证）。
- 接入 FISCO BCOS 合约与 Java SDK（Software Development Kit，软件开发工具包）调用层，完成 `Signature` / `PersonalInfo` 的 register（登记）、select（查询）和 history（历史查询），真实 VM（Virtual Machine，虚拟机）E2E 中产生 4 笔 TX（Transaction，交易）并可按区块追溯。
- 构建只读 Web/API 审计控制台，聚合 `examples/evidence/*.json` 和 `docs/evidence/*.json` 摘要，展示用户、合约、交易、查询判读、Verify/Open、揭示字段和失败场景，作为项目展示与演示入口。
- 将链上审计能力抽象到 AI 应用开发场景，可用于智能体工具调用日志、模型服务敏感操作留痕、后台审计视图和异常行为追责。

项目展示展开时少讲密码学公式，多讲“用户操作如何变成审计事件、事件如何进入控制台、异常如何追溯、哪些边界没有夸大”。

## AI 安全版本

适合对外展示 AI 安全、数据安全、可信 AI、模型服务治理、安全研发和隐私计算相关方向。重点突出安全问题抽象、身份与审计底座、异常追责和后续可扩展路径。

可写项目描述：

> 设计并实现分布式身份监管与链上审计工程原型，围绕匿名使用、行为留痕、异常追责和多方监管构建可复现证据链；进一步将该能力抽象到 AI 安全场景，用于智能体工具调用审计、模型服务敏感操作追责、数据访问留痕和隐私保护联合微调参与方审计。

可选 bullet：

- 基于 GS-TBK（Group Signatures with Time-bound Keys，带时间绑定密钥的群签名方案）和多 Node 监管流程，实现正常签名匿名、异常签名 Verify/Open 揭示的工程闭环，降低单点监管信任风险。
- 引入身份字段 CL 加密、承诺和 ZKP 处理链路，避免真实身份字段明文上链，并为审计侧提供可验证的密文与证明材料。
- 将签名、身份密文、审计查询、恶意揭示和失败场景整理为结构化证据摘要，并通过只读 Web/API 审计控制台展示，便于安全审计与项目复盘。
- 明确把项目定位为 AI 安全基础设施能力：可迁移到智能体工具调用审计、模型服务高风险操作追责、数据访问留痕和联合微调参与方身份管理。

项目展示展开时可以主动说：当前没有训练大模型，也没有实现提示词注入检测或工具调用拦截；项目价值在于“可信身份、证据留痕、异常追责和多方监管”这条安全底座。

## 不能写

以下表述不要出现在项目材料、项目标题或项目介绍里：

- 已上线生产、生产级商用身份监管平台、完整生产审计平台。
- 实现大模型训练算法、训练框架、优化器或端到端大模型微调平台。
- 实现 Prompt Injection 完整防护系统、智能体全链路安全防护网关、模型内容安全审核系统。
- 用同态加密完成大模型训练、完成全同态大模型推理或生产级 AI 治理平台。
- 在没有最终验收报告时，直接写“完整生产验收通过”；更稳的说法是“第三方功能测试用例验证通过”。
- 把 BBS+（一种选择性披露签名方案）写进核心技术栈；当前仓库和第三方测试记录都不能直接支撑它是主线实现。

更稳的替代表述：

- 生产化 smoke 验证通过的可复现工程原型。
- 分布式身份监管与链上审计原型。
- 可迁移到 AI 安全场景的身份、留痕、审计和追责底座。
- 只读 Web/API 审计控制台展示层，不连接真实链端或生产配置。
- 第三方功能测试用例验证通过，多服务器测试环境覆盖阈值密钥生成、用户注册、签名上链、异常身份揭示和隐私数据密文上链。

## 展示取舍

如果技术评审偏 AI 应用开发，就从审计控制台开始：展示 Web/API 入口、数据聚合、交易和区块证据，再讲 Rust 与 Java SDK 如何把协议和链端串起来。

如果技术评审偏 AI 安全，就从风险模型开始：为什么要匿名、为什么要可揭示、为什么不能信任单一管理员，再讲它如何迁移到智能体工具调用和模型服务敏感操作审计。

如果技术评审追问生产化程度，就直接回答：当前是工程原型，真实 VM E2E 和审计证据已经闭环，但认证授权、长期监控、告警、灾备、密钥轮换和生产安全运维仍是后续工作。

# v0.2 可审计工程原型发布说明

本文是 `v0.2-auditable-prototype` 的 release（发布）说明，面向新人交接、面试讲解和后续维护。它基于 v0.1 工程原型继续补齐失败场景、链上审计查询、恶意用户 Verify/Open（验证/揭示）、真实 VM（Virtual Machine，虚拟机）证据、JSON（JavaScript Object Notation，数据交换格式）摘要和面试讲解包。

## 当前定位

当前项目可以定位为：

> 可复现、可审计、可面试讲解的分布式身份监管工程原型。

它已经完成 Rust（系统级编程语言）多角色协议、FISCO BCOS（金融区块链合作联盟开源区块链底层平台）合约、Java SDK（Software Development Kit，软件开发工具包）调用层、身份字段 CL（Castagnos-Laguillaumie，同态加密方案）加密/ZKP（Zero-Knowledge Proof，零知识证明）、链上签名与身份密文登记、真实 VM E2E（End-to-End，端到端）smoke（冒烟验证）、审计查询、恶意揭示摘要和失败场景库。

它仍然不是可上线生产系统。生产级长期运行、监控告警、权限隔离、密钥轮换、多环境发布、灾难恢复和完整运维治理仍属于下一阶段。

## v0.2 新增能力

- 失败场景库已整理：覆盖缺合约地址、缺证书、链不可达、端口占用、错误身份字段和重复注册，记录触发命令、错误分类和恢复建议。
- 链上审计查询已形成 demo（演示）和真实 VM 复核：围绕 `Signature` 与 `PersonalInfo` 的 `select`、`history` / `selectWithBlockNumber`、TX（Transaction，交易）哈希和区块高度，证明当前可查、登记区块可追溯、登记前一区块不可追溯。
- 恶意用户揭示已形成 demo 和真实 VM 摘要：`user1` 使用 `sign_wrong`，4 个 Node（管理员节点）均从链上查到签名、Verify 失败并触发 Open；最新 `20260512T200205Z` 报告已捕获 `user_id`、`user_name` 和地址揭示字段。`user2` 作为正常对照，Verify 通过。
- 证据摘要支持 Markdown（轻量标记语言）和 JSON 输出，便于后续 Web（网页）/API（Application Programming Interface，应用程序接口）展示和 AI（Artificial Intelligence，人工智能）安全审计材料复用。
- 面试讲解包已补齐 2 分钟概述、5 分钟技术结构、架构关系、证据口径、常见追问、失败场景口径和简历描述。

### 审计查询

`scripts/evidence/run-audit-query-demo.sh` 已支持从真实 manifest 生成审计查询摘要。当前真实 VM 报告覆盖 2 个 User、2 个合约、每条记录 3 类查询：

- `select`：当前主键返回 `exists true`。
- `history@block`：登记区块返回 `ret 0`。
- `history@previous-block`：登记前一区块返回 `ret -2`。

这条证据链用于说明“用户记录在哪笔交易和哪个区块写入，以及写入前后能否追溯”。

### 恶意揭示

`scripts/evidence/run-malicious-open-demo.sh` 已支持从 E2E manifest 和 Node stdout（标准输出）/log4rs（Rust 日志框架）日志生成恶意揭示摘要。当前真实 VM 复核中：

| 用户 | 签名入口 | Verify 结果 | Open 状态 | 揭示字段 |
| --- | --- | --- | --- | --- |
| `e2e20260512T200205Z_user1` | `sign_wrong` | 失败，触发 Open | 完成 | `user_id:1`、`user_name:e2e20260512T200205Z_user1`、`127.0.0.1:60001` |
| `e2e20260512T200205Z_user2` | `sign` | 通过 | 全局完成，非本用户触发 | 不适用 |

这里可以讲“异常签名会触发揭示流程并定位到协议用户和地址”，但仍应说明当前是 smoke 级证据，不等同于生产审计平台。

### JSON 摘要

审计查询和恶意揭示脚本会同时输出 Markdown 与 JSON 摘要。JSON 文件只保留 manifest 路径与 SHA-256（安全哈希算法 256 位）、查询/日志路径、TX 哈希、区块号、Verify/Open 判读、`reveal_fields` 揭示字段和必要说明，不内嵌真实证书、私钥、账户、wallet（钱包）、keystore（密钥库）、`conf/config.toml`、runtime-state（运行时状态）或大日志。

## 最短复核路径

以下命令默认在 Ubuntu（Linux 发行版）VM 的仓库根目录执行，且真实链配置已经按安全边界准备好。

```bash
set -a
. ./.env.fisco.generated
set +a

export FISCO_CONFIG="$PWD/chain-apps/fisco-bcos-java-sdk/conf/config.toml"
export FISCO_GROUP=group0
export GRADLE_BIN=/tmp/gradle-8.10.2/bin/gradle
export GSTBK_PERSONAL_INFO_APP_DIR="$PWD/chain-apps/fisco-bcos-java-sdk"
export GSTBK_SIGNATURE_APP_DIR="$PWD/chain-apps/fisco-bcos-java-sdk"
export LD_LIBRARY_PATH="$PWD/crates/cl_encrypt:${LD_LIBRARY_PATH:-}"

bash scripts/fisco/doctor.sh
```

复核 E2E：

```bash
GRADLE_BIN=/tmp/gradle-8.10.2/bin/gradle \
  bash scripts/fisco/bootstrap-vm-smoke.sh --smoke e2e
```

基于已有 manifest 生成审计查询摘要：

```bash
bash scripts/evidence/run-audit-query-demo.sh \
  --manifest /tmp/gstbk-e2e-vm-smoke/runtime-logs/20260512T153825Z/manifest.json \
  --output-dir /tmp/gstbk-live-audit-query-20260512T153825Z
```

基于已有 manifest 生成恶意揭示摘要：

```bash
bash scripts/evidence/run-malicious-open-demo.sh \
  --manifest /tmp/gstbk-open-reveal-log-capture-runtime/runtime-logs/20260512T200205Z/manifest.json \
  --output-dir /tmp/gstbk-open-reveal-log-capture-summary-20260512T200205Z
```

## 验收 checklist

- 真实 VM E2E 报告来自 manifest，不是 fixture（夹具）样例。
- `Signature` 与 `PersonalInfo` 各有 2 笔链上写入，查询结果为 `exists true`。
- 审计查询报告覆盖 `select`、登记区块 `history` 和登记前一区块 `history`。
- 恶意揭示报告中 `user1` 是 `sign_wrong`，4 个 Node 均出现链上查询、Verify 失败、Open 完成和揭示字段。
- JSON 摘要不包含真实证书、私钥、账户、wallet、keystore、`conf/config.toml` 或 runtime-state。
- 失败场景库不提交大日志或真实敏感材料，只提交命令、分类和恢复建议。
- 面试讲解包明确说明项目是工程原型，不是完整生产系统。

## 已知限制

- 当前仍是工程原型，不是已上线生产系统。
- `gstbk-service.sh` 是本地 service supervisor（服务管理器），不是完整 systemd（Linux 系统服务管理器）或容器编排平台。
- 当前证据以真实 VM smoke、只读审计查询和摘要脚本为主，不替代长期监控、告警、权限审计或灾难恢复。
- 真实证书、账户、私钥、wallet、keystore 和 `conf/config.toml` 仍需在 VM 或本机安全目录准备，不进入 Git（分布式版本控制系统）。
- 审计查询和恶意揭示摘要适合支撑演示与面试讲解，后续若对外展示仍需要 Web/API 层、权限控制、脱敏策略和更完整的日志留存策略。

## 下一阶段候选方向

- Web/API 展示：把 E2E manifest、审计查询、恶意揭示和 JSON 摘要做成只读演示界面或服务接口。
- Open 揭示日志增强：统一 stdout、log4rs 文件日志和结构化 JSON 事件，减少人工判读成本。
- AI 安全衔接：把“匿名使用、行为留痕、异常追责”的证据链抽象到智能体工具调用、模型服务审计或敏感操作追责场景。
- 生产级运维：推进 systemd/容器化、多环境配置分层、证书轮换、监控告警、自动恢复和最小权限策略。

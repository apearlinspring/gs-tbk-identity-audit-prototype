# v0.1 工程原型发布与复现说明

本文是 `v0.1-engineering-prototype` 的 release（发布）与复现入口，面向新人、技术评审和后续维护者。目标是不用翻聊天记录，也能判断当前项目完成到什么程度、如何复现、如何验收，以及哪些能力还不属于生产系统。

## 当前定位

当前项目定位为：生产化 smoke（冒烟验证）通过的分布式身份监管工程原型。

它已经从“资料包 + 多个 demo（演示工程）快照”整理为可复现工程：Rust（系统级编程语言）workspace（工作区）、FISCO BCOS（金融区块链合作联盟开源区块链底层平台）合约、Java SDK（Software Development Kit，软件开发工具包）调用层、身份字段加密、运行脚本、服务管理和证据文档已经收口。

它还不是完整生产系统。长期运行、监控告警、安全配置治理、权限隔离、多环境发布、部署自动化和生产级故障恢复仍需后续建设。

当前复现基线复用以下链上合约地址：

| 合约 | 地址 | 部署区块 |

`PersonalInfo` 保存身份密文、承诺和 ZKP（Zero-Knowledge Proof，零知识证明）材料；`Signature` 保存用户签名 JSON（JavaScript Object Notation，数据交换格式）。

## 关键能力

- CL（Castagnos-Laguillaumie，同态加密方案）身份字段加密、证明生成和验证。
- ZKP 身份字段证明随身份密文生成，并可由本地流程验证。
- 用户匿名签名 JSON 上链，链上 `Signature` 可查询。
- 身份密文、承诺和证明材料上链，链上 `PersonalInfo` 可查询。
- 1 Proxy + 4 Node + 2 User 多角色流程完成 KeyGen（联合密钥生成）、Join（用户加入）、Revoke（撤销）、Sign（签名）、Verify（验证）和 Open（揭示）。
- `run-e2e.sh` 支持 runtime（运行时）配置隔离、manifest（运行清单）、端口预检、失败尾日志和清理。
- `gstbk-service.sh` 支持本地服务管理：`start`、`stop`、`status`、`restart` 和 `tail`。
- `generate-e2e-report.sh` 可从 manifest 和日志生成 Markdown（轻量标记语言）验收报告，覆盖成功和 `success false` 失败场景。

## 最短复现路径

以下路径默认在 Ubuntu VM 的仓库根目录执行，且真实链节点已经存在。完整说明见 [运行入口](../operations/README.md)、[FISCO BCOS 运维自动化 Runbook](../operations/fisco-runbook.md) 和 [本地服务管理](../operations/service-supervision.md)。

### 1. 准备 `.env`

```bash
cp .env.example .env
set -a
. ./.env
set +a
export LD_LIBRARY_PATH="$PWD/crates/cl_encrypt:${LD_LIBRARY_PATH:-}"
export GSTBK_PERSONAL_INFO_APP_DIR="$PWD/chain-apps/fisco-bcos-java-sdk"
export GSTBK_SIGNATURE_APP_DIR="$PWD/chain-apps/fisco-bcos-java-sdk"
```

### 2. 准备 FISCO SDK 配置

真实 `conf/config.toml`、SDK 证书、账户和私钥只放在 VM 或本机安全目录，不提交 Git。

```bash
bash scripts/fisco/prepare-sdk-conf.sh \
  --node-sdk-dir /home/gstbk/fisco/nodes/127.0.0.1/sdk \
  --app-dir "$PWD/chain-apps/fisco-bcos-java-sdk" \
  --group group0 \
  --peers 127.0.0.1:20200,127.0.0.1:20201 \
  --force

export FISCO_CONFIG="$PWD/chain-apps/fisco-bcos-java-sdk/conf/config.toml"
export FISCO_GROUP=group0
export FISCO_CONSOLE_DIR=/home/gstbk/fisco/console
export GRADLE_BIN=/tmp/gradle-8.10.2/bin/gradle
```

### 3. 运行 doctor

已有合约地址时：

```bash
export GSTBK_PERSONAL_INFO_CONTRACT_ADDRESS=0x6546c3571f17858ea45575e7c6457dad03e53dbb
export GSTBK_SIGNATURE_CONTRACT_ADDRESS=0xcceef68c9b4811b32c75df284a1396c7c5509561

bash scripts/fisco/doctor.sh
```

首次部署或还没有合约地址时：

```bash
bash scripts/fisco/doctor.sh --allow-missing-contract-addresses
```

### 4. 部署或复用合约

优先复用当前基线合约：

```bash
bash scripts/fisco/deploy-contracts.sh \
  --mode reuse \
  --personal-info-address 0x6546c3571f17858ea45575e7c6457dad03e53dbb \
  --signature-address 0xcceef68c9b4811b32c75df284a1396c7c5509561

set -a
. ./.env.fisco.generated
set +a
```

如果确实需要新部署：

```bash
bash scripts/fisco/deploy-contracts.sh --mode deploy
```

### 5. 运行 E2E 或服务管理 smoke

推荐的一键 VM bootstrap 验收入口：

```bash
GRADLE_BIN=/tmp/gradle-8.10.2/bin/gradle \
  bash scripts/fisco/bootstrap-vm-smoke.sh --smoke e2e
```

一键 E2E 复现：

```bash
bash scripts/run-local/run-e2e.sh \
  --users 2 \
  --nodes 4 \
  --runtime-dir /tmp/gstbk-e2e-smoke \
  --reuse-chain \
  --contract-addresses-from-env \
  --timeout-seconds 300
```

生成本轮 E2E 验收报告：

```bash
bash scripts/evidence/generate-e2e-report.sh \
  --manifest /tmp/gstbk-e2e-smoke/runtime-logs/<timestamp>/manifest.json
```

当前正式 VM 报告为 `docs/evidence/e2e-report-20260512T153825Z.md`，对应真实 manifest `/tmp/gstbk-e2e-vm-smoke/runtime-logs/20260512T153825Z/manifest.json`；它记录合约地址、区块 `50` -> `54`、4 笔 TX 哈希、角色阶段、耗时 `411` 秒和日志 SHA-256。

本地服务管理 smoke：

```bash
bash scripts/run-local/gstbk-service.sh start all
bash scripts/run-local/gstbk-service.sh status all
bash scripts/run-local/gstbk-service.sh tail proxy --lines 80
bash scripts/run-local/gstbk-service.sh stop all
```

## 验收 checklist

- `git status --short --branch` 只显示当前分支行，或只显示本轮预期文档改动。
- `bash scripts/fisco/doctor.sh` 通过，并能读取当前 `blockNumber`。
- `LD_LIBRARY_PATH="$PWD/crates/cl_encrypt:${LD_LIBRARY_PATH:-}" cargo check --workspace --locked` 通过。
- `bash -n scripts/run-local/*.sh scripts/fisco/*.sh` 通过。
- `run-e2e.sh` 输出的 `manifest.json` 中 `success` 为 `true`，`config.mode` 为 `runtime`，且 `config.legacy_fixture_configs` 为 `false`。
- `generate-e2e-report.sh` 可基于本轮 `manifest.json` 生成 `docs/evidence/e2e-report-<timestamp>.md`，报告包含命令、合约地址、区块高度、TX（Transaction，交易）哈希、Role（角色）阶段、日志路径和 SHA-256（安全哈希算法 256 位）。
- 真实 VM bootstrap E2E 报告应明确来自 `bootstrap-vm-smoke.sh --smoke e2e` 的真实 manifest，而不是 fixture 样例；当前正式报告为 `docs/evidence/e2e-report-20260512T153825Z.md`。
- `gstbk-service.sh status all` 在启动后显示 Proxy、4 个 Node 和 2 个 User 运行中；停止后均为 `stopped`。
- Proxy 和 4 个 Node 日志出现 KeyGen 完成；2 个 User 日志出现 Join 和 Sign 完成。
- 4 个 Node 日志出现 `Signature query stdout: exists true`，并完成 Verify/Open。
- `Signature` 和 `PersonalInfo` 均出现链上交易哈希与区块高度，查询结果为 `exists true`。
- 运行后 `50000`、`50001` 到 `50004`、`60001`、`60002` 无监听残留。
- 运行后 Git 工作树不出现真实 `conf/config.toml`、证书、账户、wallet、keystore、运行日志或 runtime state（运行时状态）文件。

## 已知限制

- 当前是工程原型，不是已上线生产系统。
- 真实证书、账户、私钥、wallet、keystore 和 `conf/config.toml` 不入库，需要在 VM 或本机安全目录准备。
- CI（Continuous Integration，持续集成）不持有真实链配置，也不跑真实 FISCO BCOS 链端 E2E；链端闭环仍以 VM smoke 证据为准。
- `gstbk-service.sh` 是本地 service supervisor，不是 systemd（Linux 系统服务管理器）、容器平台或生产编排系统。
- E2E 报告生成器只解析已有 manifest 和日志，不跑真实链、不采集新链上数据，也不替代长期审计平台。
- legacy fixture（历史夹具）配置仍保留为兼容路径，新复现应优先使用 runtime 配置目录。
- 当前只内置 `examples/id-info/user1.json` 和 `examples/id-info/user2.json`；扩展到更多 User 时需要准备独立脱敏身份输入或身份密文。
- 生产级监控告警、权限隔离、密钥轮换、审计查询平台化和多环境发布还在后续计划中。

## 项目讲解口径

可以这样概括：这个项目最初是资料包和多个历史 demo 快照，我把它整理成了基于 Rust + FISCO BCOS 的分布式身份监管工程原型。现在项目已经完成链端合约、Java SDK 调用层、Rust 多角色 E2E、CL 身份字段加密/ZKP、签名和身份密文上链、本地服务管理以及运行证据沉淀。

讲能力边界时要明确：它是生产化 smoke 验证通过的原型系统，不是已上线生产系统。后续方向是安全配置治理、CI 收紧、部署自动化和更完整的审计查询/展示能力，让系统继续向可复现、可运维、可审计和可交接演进。

# 分布式身份监管项目与 AI（Artificial Intelligence，人工智能）安全方向衔接说明

本文用于项目解释、技术追问和毕设选题讨论。核心口径是：当前项目不是 AI（Artificial Intelligence，人工智能）大模型项目，也不是模型训练算法；它更适合作为 AI 安全场景里的身份、留痕、审计和追责前置能力。

一句话可以这样讲：

> 我现在的分布式身份监管项目，解决的是“匿名使用、行为留痕、异常可追责、多方共同监管”的工程问题。这个能力可以自然迁移到 AI 安全里的智能体工具调用审计、模型服务敏感操作追责、数据访问留痕和联合微调参与方审计，但不能夸大成已经实现了大模型训练或 Prompt Injection（提示词注入）防护。

## 当前项目的可迁移能力

当前项目基于 Rust（系统级编程语言）和 FISCO BCOS（金融区块链合作联盟开源区块链底层平台）实现分布式身份监管工程原型，已经在真实 VM（Virtual Machine，虚拟机）上完成 1 个 Proxy（代理）、4 个 Node（管理员节点）、2 个 User（用户）的 E2E（End-to-End，端到端）生产化 smoke（冒烟验证）。这些能力可以抽象成 AI 安全方向的基础设施能力。

| 当前能力 | 在原项目中的含义 | 迁移到 AI 安全后的含义 |
| --- | --- | --- |
| 匿名使用 | User 使用 GS-TBK（Group Signatures with Time-bound Keys，带时间绑定密钥的群签名方案）完成群签名，正常情况下不直接暴露具体身份 | 用户、智能体或服务账号可以在最小暴露身份的前提下调用模型或工具 |
| 行为留痕 | 签名、身份密文和证明材料写入链上合约 | 模型调用、工具调用、数据访问、训练任务和评测事件形成可查询记录 |
| 链上审计 | `Signature` 和 `PersonalInfo` 支持当前查询与按区块历史查询 | 审计方可以追溯某次敏感操作是否发生、何时发生、由哪个审计身份关联 |
| 异常追责 | Verify/Open（验证/揭示）流程能在异常签名时定位协议用户和地址 | 当智能体越权调用工具、模型服务执行敏感操作或训练参与方违规时，有追责入口 |
| 多方监管 | 多个 Node 持有监管材料并参与验证和揭示，减少单点信任 | 企业、安全团队、合规方或平台方共同参与监管，不把审计权集中给单一系统管理员 |
| 隐私字段保护 | 身份字段先做 CL（Castagnos-Laguillaumie，同态加密方案）加密和 ZKP（Zero-Knowledge Proof，零知识证明），再登记证据 | 敏感身份、数据摘要或任务元数据可以先做加密、承诺或证明，再进入审计流程 |

这条迁移路径的关键是“能力抽象”，不是“项目改名”。原项目里审计的是分布式身份和签名事件；AI 安全场景里可以审计模型调用、工具调用、训练任务和敏感数据访问事件。

## 展示入口如何讲

项目展示或答辩时，建议把只读 Web（网页）/API（Application Programming Interface，应用程序接口）审计控制台作为第一展示入口。它从 `examples/evidence/*.json` 和 `docs/evidence/*.json` 读取已提交摘要，展示 User（用户）、Contract（合约）、TX（Transaction，交易）、区块、查询判读、Verify/Open（验证/揭示）、揭示字段和失败场景。

这层展示对 AI 安全叙事很关键：它把“身份、留痕、追责”从抽象描述变成可浏览、可查询、可复盘的证据视图。但也要同步说明边界：控制台不连接真实 FISCO BCOS 节点，不读取证书、私钥、wallet（钱包）、keystore（密钥库）、`conf/config.toml` 或生产大日志，只是演示与项目讲解用的只读展示层。

## 适合衔接的 AI 安全场景

### 智能体工具调用审计

Agent（智能体）在调用搜索、数据库、代码执行、支付、文件读写等工具时，往往需要回答三个问题：谁发起、调用了什么、结果是否可追溯。当前项目的身份与链上审计思路可以迁移为：

- 为智能体、用户或服务账号分配可审计身份。
- 对高风险工具调用生成签名或审计事件。
- 将工具名称、请求摘要、结果摘要、时间和调用方身份承诺写入链上或可信日志。
- 正常情况下隐藏真实个人身份，异常时由多方监管流程定位责任主体。

这里能讲“工具调用审计”和“异常追责”，不要讲成已经具备完整智能体安全网关。

### 模型服务敏感操作追责

模型服务可能触发发送邮件、删除数据、修改配置、发起交易、发布内容等敏感操作。当前项目可迁移的点是：

- 敏感操作执行前要求调用方产生可验证身份凭证。
- 操作请求和审批结果形成链上审计锚点。
- 异常操作不只依赖本地日志，而是能结合链上记录追溯。
- 多个监管节点参与揭示或确认，降低单点管理员滥用风险。

这更接近“模型应用安全”和“AI 系统审计”，不是模型结构安全。

### 数据访问留痕

AI 应用经常接入知识库、用户画像、业务数据库和训练样本库。当前项目的链上留痕能力可以迁移为：

- 记录某个模型服务、用户或智能体访问了哪类数据资源。
- 只上链数据摘要、数据承诺、访问策略编号和任务编号，不上链原始数据。
- 对敏感身份字段使用加密、承诺或证明，避免审计系统本身变成隐私泄露点。
- 支持“当前能查到”和“登记区块可追溯”的证据链。

这个方向适合写成“数据访问审计”和“敏感操作留痕”，不要写成“已经实现数据泄露检测平台”。

### 联合微调参与方身份与审计

Federated Fine-tuning（联合微调）或多方协作微调中，参与方身份、数据摘要、模型版本、训练轮次和评测结果都需要可信记录。当前项目可以作为毕设方向的前置能力：

- 用分布式身份机制管理训练参与方。
- 用链上合约记录训练任务、参与方、数据摘要、模型版本和评测摘要。
- 用身份字段加密、承诺和证明思路保护敏感元数据。
- 在异常提交、违规数据使用或结果争议时提供审计追溯依据。

这里的合理表述是“为隐私保护联合微调提供身份与审计基础”，不是“用 GS-TBK 直接训练大模型”。

更具体地说，当前项目完成的是联合微调之前的身份与链上审计前置能力：参与方身份、审计事件、event schema（事件模式）、Web（网页）/API（Application Programming Interface，应用程序接口）展示和异常追责。真正的联合微调 demo（演示工程）应作为毕设新增实现，不要把现有仓库表述为已经完成模型训练。

## 不适合硬蹭的部分

为了避免项目材料和项目展示里被追问穿，下面这些边界要主动说清楚。

| 不应夸大的说法 | 为什么不准确 | 更准确的说法 |
| --- | --- | --- |
| 这是一个大模型项目 | 当前主线没有模型训练、推理服务或大模型 API（Application Programming Interface，应用程序接口）调用 | 这是可迁移到 AI 安全场景的身份监管与链上审计工程原型 |
| 实现了模型训练算法 | 当前项目实现的是身份、群签名、链上登记和审计流程，不是优化器、训练框架或参数更新算法 | 后续毕设可在它之上设计联合微调原型 |
| 实现了大模型安全评测框架 | 当前没有覆盖训练数据复述、敏感字段泄露、成员推断等模型评测流程 | 当前项目提供审计证据基础，评测框架属于后续扩展 |
| 实现了 Prompt Injection 防护系统 | 当前没有做提示词注入检测、工具调用策略拦截或上下文隔离 | 可以服务于提示词注入事件后的留痕和追责，但不是防护算法本身 |
| 完成了生产级 AI 治理平台 | 当前是生产化 smoke 通过的工程原型，不是完整生产系统 | 可以说完成真实 VM 多角色 E2E、链上审计查询和异常揭示摘要 |
| 用同态加密完成大模型训练 | 当前 CL 同态加密用于身份字段加密和证明，不适合直接表述为大模型全流程加密训练 | 可以说复用了敏感字段保护、承诺和证明思路 |

讲解中最稳的讲法是：我没有把这个项目包装成大模型系统，而是把它沉淀为 AI 安全里的“身份可信、操作留痕、异常追责、链上审计”底座。

项目材料中尤其不能写这三类硬话：

- 已上线生产、生产级商用身份监管平台、完整生产审计平台。
- 实现大模型训练算法、训练框架或端到端大模型微调平台。
- 实现 Prompt Injection 完整防护系统、智能体全链路安全防护网关。

## 与毕设方向的衔接

建议毕设方向继续沿用 [面向大模型联合微调的隐私保护与可信审计机制研究](plans/thesis-ai-security-finetuning-plan.md) 这条主线。当前项目在其中扮演“身份与链上审计前置能力”的角色，后续毕设实现才进入联合微调和模型安全评测。

可以拆成六个复用点：

| 复用点 | 当前项目提供什么 | 毕设需要新增什么 |
| --- | --- | --- |
| 身份 | Proxy、Node、User 的分布式身份监管流程，多节点参与 Verify/Open | 将训练参与方、模型服务账号和审计角色映射到身份体系 |
| 审计 | FISCO BCOS 合约登记、当前查询、按区块历史查询、TX（Transaction，交易）证据 | 新增训练任务、轮次提交、聚合完成和评测完成等审计事件 |
| 事件 schema | `events[]` 统一抽象审计查询、恶意揭示和失败场景 | 扩展 `training_task_created`、`round_update_submitted`、`evaluation_completed` 等训练事件 |
| Web/API 展示 | 只读 `audit-console` 聚合证据摘要，不读取真实链端秘密 | 展示训练任务、参与方、数据摘要、指标摘要和安全评测摘要 |
| 异常追责 | Verify/Open 能在异常签名时定位责任主体 | 面向异常提交、违规数据使用、指标争议或审计争议提供追责入口 |
| 隐私材料保护 | 身份字段 CL 加密、ZKP 和敏感材料不上明文链 | 对训练数据摘要、参与方提交、评测结果做哈希、承诺、差分隐私或安全聚合 |

毕设最小可行路线建议收敛为四步：

1. 明文联合微调 baseline（基线）：先跑通单方和两方明文微调，形成可复现实验指标。
2. 审计事件上链/摘要：记录训练任务、参与方、数据摘要、模型版本、轮次提交和评测摘要。
3. 隐私增强：选择 DP（Differential Privacy，差分隐私）或 Secure Aggregation（安全聚合），不要强行把 CL 同态加密用于完整模型训练。
4. 安全评测：覆盖训练数据复述、敏感字段泄露、成员推断或 Prompt Injection 诱导泄露，并与 baseline 对比。

推荐毕设表述：

> 本科项目阶段已经完成分布式身份监管与链上审计工程原型，定位为身份与链上审计前置能力。毕设将在此基础上新增联合微调 demo：训练过程由常规 AI 训练框架完成，项目重点研究参与方身份管理、训练事件上链/摘要、数据摘要保护、差分隐私或安全聚合增强，以及模型隐私风险评测。

这句话的好处是边界清楚：训练归训练，审计归审计，当前项目不是被强行改造成大模型项目，而是成为可信审计基础。

## 技术追问口径

**问：这个项目和 AI 有什么关系？**

答：它本身不是 AI 项目，我不会强行说它训练了大模型。它解决的是身份可信和审计追责问题，而 AI 安全里很多风险也落在这个层面，比如智能体调用工具、模型服务访问敏感数据、联合微调参与方提交更新。我的衔接方式是把现有项目作为身份与审计底座，再在毕设里叠加联合微调、隐私保护和模型安全评测。

**问：你做的是不是 Prompt Injection 防护？**

答：不是。Prompt Injection 防护需要检测恶意提示词、隔离上下文、限制工具权限和拦截危险调用。当前项目更适合做事件后的可信留痕和追责：一旦智能体发生越权工具调用，可以追溯调用方、调用摘要、时间和链上证据。

**问：你有没有做模型训练？**

答：当前项目没有做模型训练。它的主线是 Rust 多角色身份监管、FISCO BCOS 链上审计、身份字段加密和异常揭示。后续毕设计划会新增联合微调 baseline、隐私增强训练和安全评测，把当前项目作为参与方身份和训练审计模块。

**问：为什么这能算 AI 安全相关？**

答：AI 安全不只有模型算法本身，也包括模型服务和智能体系统的安全治理。真实落地时，谁能访问数据、谁触发了敏感工具、某次训练由哪些参与方和数据摘要构成，这些都需要身份、留痕、审计和追责。当前项目正好提供这类底层能力。

**问：最大的边界是什么？**

答：边界有三个：第一，它不是大模型训练算法；第二，它不是大模型安全评测框架；第三，它不是 Prompt Injection 防护系统。更准确地说，它是一个可以迁移到 AI 安全场景的分布式身份监管与链上审计工程原型。

## 推荐最终口径

项目材料或答辩中可以使用下面这段：

> 当前项目是基于 Rust + FISCO BCOS 的分布式身份监管与链上审计工程原型，已经完成多角色 E2E、身份密文/ZKP 处理、签名与身份记录上链、链上历史查询、事件摘要、只读 Web/API 展示和异常签名揭示摘要。它不是大模型项目，但它沉淀了 AI 安全可复用的身份与审计前置能力：匿名使用、行为留痕、异常追责和多方监管。后续毕设会新增联合微调 demo，按“明文 baseline -> 审计事件上链/摘要 -> 差分隐私或安全聚合 -> 安全评测”的路线迁移到隐私保护联合微调场景。

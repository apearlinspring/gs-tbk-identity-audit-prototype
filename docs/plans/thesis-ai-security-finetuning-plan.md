# 毕设设计方案：面向 AI（Artificial Intelligence，人工智能）安全的联合微调隐私保护与可信审计

## 题目建议

首选：

> 面向大模型联合微调的隐私保护与可信审计机制研究

备用：

> 基于隐私增强与链上审计的两方联合微调原型系统

## 定位刷新

当前 GS-TBK（Group Signatures with Time-bound Keys，带时间绑定密钥的群签名方案）分布式身份监管项目，定位为“身份与链上审计前置能力”，不是已经完成的联合微调系统。它已经沉淀了参与方身份、链上留痕、事件摘要、Web（网页）/API（Application Programming Interface，应用程序接口）只读展示和异常追责能力；后续毕设实现才进入 Federated Fine-tuning（联合微调）demo（演示工程）。

毕设主线应明确分成两层：

- 已有基础：分布式身份监管、FISCO BCOS（金融区块链合作联盟开源区块链底层平台）链上审计、`events[]` event schema（事件模式）、只读 `audit-console`（审计控制台）和 Verify/Open（验证/揭示）追责流程。
- 新增实现：明文联合微调 baseline（基线）、训练审计事件上链或生成摘要、DP（Differential Privacy，差分隐私）或 Secure Aggregation（安全聚合）隐私增强、安全评测与实验对比。

这条路线不强行把 CL（Castagnos-Laguillaumie，同态加密方案）同态加密用于完整模型训练。CL 更适合复用在敏感身份字段、数据摘要承诺、可验证登记和审计材料保护上；模型训练本身优先使用常规 AI 训练框架与轻量隐私增强机制完成。

## 问题定义

两方各自持有私有训练数据，希望共同微调一个公开模型或开源小模型。系统需要回答四类问题：

- 训练流程是否能先用明文数据跑通，并形成可复现实验 baseline。
- 训练任务、参与方身份、数据摘要、模型版本、训练轮次和评测结果是否能形成链上或摘要化审计证据。
- 在不直接上链原始数据、不把敏感样本写入日志的前提下，是否能引入 DP 或安全聚合降低泄露风险。
- 微调后的模型是否存在训练样本复述、敏感字段泄露、成员推断或 Prompt Injection（提示词注入）诱导泄露等安全风险。

## 最小可行路线

毕设最小可行路线固定为：

> 明文联合微调 baseline -> 审计事件上链/摘要 -> 隐私增强（DP 或安全聚合）-> 安全评测。

### 阶段 1：明文联合微调 baseline

目标：先把两方联合微调流程跑通，得到可复现实验基线，不在第一阶段引入复杂密码学。

任务：

- 选择小模型和小数据集，例如 BERT（Bidirectional Encoder Representations from Transformers，双向编码器表征模型）、TinyLlama（轻量级 Llama 模型）或 Qwen（通义千问模型）小参数版本。
- 将公开或脱敏数据拆分为 `data_a` 和 `data_b`，模拟两方持有数据。
- 实现单方微调、两方明文合并数据微调和统一训练入口。
- 记录训练配置、模型版本、数据摘要、准确率、损失、训练时长和资源占用。

输入：

- 公开模型名称、模型版本和初始权重摘要。
- 脱敏训练数据 `data_a`、`data_b` 及其 SHA-256（安全哈希算法 256 位）摘要。
- 训练配置，例如 epoch（训练轮次）、batch size（批大小）、learning rate（学习率）和随机种子。

输出：

- baseline 模型 checkpoint（检查点）或可复现实验脚本。
- 训练 manifest（运行清单），记录模型、数据摘要、配置、指标和运行环境。
- 明确函数抽象：`fine_tune(model, data_a, data_b, config) -> fine_tuned_model`。

验收指标：

- 单方微调和两方明文联合微调均可复现。
- 至少记录准确率、损失、训练时长和内存或显存占用。
- baseline 阶段不宣称隐私保护，只作为后续对比基准。

风险边界：

- 使用公开、脱敏或合成数据，不处理真实身份原始样本。
- 明文 baseline 是实验起点，不代表隐私保护方案已经完成。
- 训练规模控制在本机或 VM（Virtual Machine，虚拟机）可复现范围内。

### 阶段 2：审计事件上链/摘要

目标：把训练过程关键元数据转化为可查询、可展示、可追溯的审计事件，优先复用当前项目的身份、链上审计和事件摘要能力。

任务：

- 设计训练审计事件：`training_task_created`、`participant_joined`、`round_update_submitted`、`aggregation_completed`、`evaluation_completed`。
- 为每个事件记录 `task_id`、`participant_id`、`model_name`、`model_version`、`data_commitment`、`metric_digest`、`timestamp` 和 `status`。
- 选择最小落地方式：先生成 `events[]` 摘要，再将关键事件写入 FISCO BCOS；或在 VM 环境可用时直接写入训练审计合约。
- 让只读 Web/API 审计控制台能够展示训练任务、参与方、数据摘要、指标摘要和证据引用。

输入：

- 阶段 1 的训练 manifest、数据摘要和指标摘要。
- 当前项目已有的参与方身份、链端配置模板和 `events[]` schema。
- 可选训练审计合约，例如 `TrainingTask.sol` 和 `TrainingAudit.sol`。

输出：

- 训练任务审计摘要 JSON（JavaScript Object Notation，数据交换格式）。
- 可选链上交易哈希、区块号和合约查询结果。
- Web/API 可消费的训练审计事件模型。

验收指标：

- 可按 `task_id` 查询训练任务摘要。
- 审计事件能回指训练 manifest、数据摘要、指标摘要和链上证据。
- `/api/evidence` 或后续训练审计 API 能展示关键事件，不读取真实证书、私钥、wallet（钱包）、keystore（密钥库）或 `conf/config.toml`。
- 上链内容只包含哈希、承诺、状态和摘要，不包含原始训练样本、完整模型参数或真实身份材料。

风险边界：

- 链上审计证明“某次摘要被登记”，不直接证明训练过程完全可信。
- 摘要化事件是审计锚点，不替代原始实验记录、代码版本和运行环境复核。
- 训练审计合约应保持最小字段，避免把链作为隐私数据存储层。

### 阶段 3：隐私增强（DP 或安全聚合）

目标：在 baseline 之上引入一个可解释、可评测的轻量隐私增强机制，而不是追求完整加密训练。

默认路线二选一，时间允许时可都做简化版：

- DP：对梯度、样本级贡献或统计量做裁剪与加噪，记录 privacy budget（隐私预算）参数。
- Secure Aggregation：模拟两方提交被掩码保护的更新，使聚合方只看到聚合结果。

任务：

- 将阶段 1 的训练入口扩展出 `privacy_mode`，至少支持 `none` 和 `dp` 或 `secure_aggregation`。
- 记录隐私参数、聚合轮次、提交摘要和聚合结果摘要。
- 将隐私增强模式和关键参数写入阶段 2 的训练审计事件摘要。
- 与明文 baseline 对比效果、训练时长和安全收益。

输入：

- 阶段 1 的训练脚本和 baseline 指标。
- DP 参数，例如 clipping norm（裁剪范数）、noise multiplier（噪声倍率）、epsilon（隐私预算 ε）和 delta（隐私预算 δ）；或安全聚合的掩码、轮次和聚合配置。
- 阶段 2 的训练审计事件 schema。

输出：

- 隐私增强模型或可复现实验脚本。
- 隐私参数摘要、指标摘要和审计事件。
- 与 baseline 的效果、成本和安全性对比表。

验收指标：

- 至少一种隐私增强模式能跑通并产生可复现实验结果。
- 能展示隐私增强前后的准确率、损失、训练时长和资源成本差异。
- 审计摘要能标注本次训练使用的隐私模式和参数摘要。
- 不把 CL 同态加密包装成完整梯度或模型训练方案。

风险边界：

- DP 参数需要说明实验性质，不能声称达到生产级隐私保护。
- 两方安全聚合 demo 可以是教学级模拟，不宣称覆盖掉线、恶意参与方和大规模客户端。
- 隐私增强机制只降低特定泄露风险，不保证模型完全不泄露训练数据。

### 阶段 4：安全评测

目标：证明毕设不是只“会微调”，而是能围绕 AI 安全给出可运行或可复核的风险评测。

任务：

- 设计训练数据复述测试，观察模型是否输出训练样本片段。
- 设计敏感字段提取测试，观察模型是否泄露身份、联系方式、编号等字段。
- 设计成员推断或近似成员推断样例，比较训练内外样本的输出差异。
- 设计 Prompt Injection 诱导泄露测试，评估模型是否被诱导输出训练数据或敏感摘要。
- 对比 baseline 与隐私增强版本的泄露风险、任务效果和成本。

输入：

- baseline 模型、隐私增强模型和对照模型。
- 脱敏安全评测集、攻击提示词、成员推断样例和敏感字段规则。
- 阶段 2/3 生成的审计事件和指标摘要。

输出：

- AI 安全评测报告。
- 风险矩阵：风险、触发条件、影响、观测结果、缓解方案。
- 评测事件摘要或链上指标摘要。

验收指标：

- 至少覆盖训练数据复述、敏感字段泄露和 Prompt Injection 诱导泄露三类测试。
- 能给出 baseline 与隐私增强版本的对比结论。
- 评测结果能形成摘要，回写到训练审计事件或生成可展示 JSON。
- 报告明确未覆盖的风险，例如生产级红队测试、复杂成员推断、模型逆向和恶意参与方攻击。

风险边界：

- 安全评测只能说明测试集和攻击样例下的表现，不能证明模型绝对安全。
- 小模型、小数据集上的结论不能直接外推到生产大模型。
- Prompt Injection 测试是泄露诱导评估，不等同于完整防护系统。

## 当前项目复用点

| 复用点 | 当前已有能力 | 毕设中的用法 |
| --- | --- | --- |
| 身份 | Proxy（代理）、Node（管理员节点）、User（用户）和多方 Verify/Open 流程 | 将训练参与方、审计方和模型服务账号映射为可审计主体 |
| 审计 | FISCO BCOS 合约写入、当前查询、历史查询、TX（Transaction，交易）和区块证据 | 记录训练任务、参与方加入、轮次提交、聚合完成和评测完成 |
| 事件 schema | `events[]` 把审计查询、恶意揭示和失败场景统一为机器可读事件 | 扩展训练事件类型，服务论文图表、Web/API 和实验报告 |
| Web/API 展示 | `audit-console` 只读展示证据摘要，不读取真实链端秘密 | 展示训练任务、数据摘要、指标摘要、风险评测和证据引用 |
| 异常追责 | 异常签名 Verify 失败后可触发 Open 揭示责任主体 | 面向异常提交、违规数据使用、指标争议或审计争议提供追责入口 |
| 隐私字段保护 | 身份字段 CL 加密、ZKP（Zero-Knowledge Proof，零知识证明）和敏感材料不上明文链 | 用于身份字段、数据摘要承诺或可验证登记，不用于完整模型训练 |

## 合约与事件建议

最小合约或摘要模型可以先覆盖两个对象：

- `TrainingTask`：记录任务编号、模型名称、模型版本、参与方集合、数据摘要集合、状态和创建时间。
- `TrainingAudit`：记录训练开始、参与方提交、聚合完成、评测完成、异常标记和指标摘要。

建议训练事件字段：

| 字段 | 说明 |
| --- | --- |
| `task_id` | 训练任务编号，由任务配置和创建时间派生。 |
| `event_type` | 训练事件类型，例如 `training_task_created`、`round_update_submitted`。 |
| `participant_id` | 参与方身份编号或承诺。 |
| `model_name` | 模型名称。 |
| `model_version` | 模型版本、权重摘要或代码版本。 |
| `data_commitment` | 数据摘要、哈希或承诺。 |
| `metric_digest` | 指标摘要，不包含完整日志。 |
| `privacy_mode` | `none`、`dp` 或 `secure_aggregation`。 |
| `evidence_refs` | manifest、指标文件、链上交易和评测报告引用。 |
| `status` | `created`、`submitted`、`aggregated`、`evaluated`、`failed` 等状态。 |

## 论文产出

- 一个两方联合微调 baseline demo。
- 一套训练审计事件摘要和可选链上合约调用。
- 一个 DP 或安全聚合隐私增强版本。
- 一组 AI 安全评测样例和报告。
- 一套实验对比：
  - 单方微调。
  - 明文两方联合微调。
  - 带审计事件的联合微调。
  - 隐私增强联合微调。
  - 安全评测与成本对比。

## 时间安排

- 第 1-2 周：完成数据脱敏、模型选择、单方与两方明文 baseline。
- 第 3-4 周：定义训练 manifest、训练事件摘要和链上字段，打通审计事件展示。
- 第 5-6 周：实现 `TrainingTask` / `TrainingAudit` 或摘要生成器，并完成查询或 Web/API 展示。
- 第 7-8 周：实现 DP 或安全聚合版本，补齐隐私参数和指标摘要。
- 第 9-10 周：完成训练数据复述、敏感字段泄露、成员推断或 Prompt Injection 安全评测。
- 第 11-12 周：整理论文、图表、答辩材料和项目表述。

## 项目材料表达建议

可写：

> 设计面向联合微调的隐私保护与可信审计原型路线，复用 Rust + FISCO BCOS 分布式身份监管项目中的参与方身份、事件 schema、Web/API 审计展示和异常追责能力；后续实现明文联合微调 baseline、训练事件上链/摘要、差分隐私或安全聚合增强，以及模型隐私泄露评测。

谨慎写：

> 基于同态加密完成大模型训练。

除非后续确实实现完整加密训练并有可复现实验，否则不要这样写。当前最稳妥的表述是：CL 同态加密用于敏感身份字段、摘要承诺和审计材料保护，不用于完整模型训练。
